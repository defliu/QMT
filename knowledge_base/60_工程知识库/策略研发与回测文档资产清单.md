## 二、每个策略必产的文档

### 2.1 策略项目目录

```
Project_XX_策略名\
├── 项目总览与下一步.md       ← 策略背景、当前状态、待办事项
├── SPEC_策略名.md            ← 策略设计规格（Hermes评审用）
└── 其他                     ← 调研脚本、中间数据、验证截图
```

**规则**：
- `Project_` 编号递增，`XX` 是两位数字
- 名称用中文简写，如 `Project_13_策略生产加固与择时落地`
- `项目总览与下一步.md` 必须包含：背景、目标、当前状态、待办列表、阻塞项

### 2.2 策略卡片（知识库 → 30_策略卡片）

```
knowledge_base/30_策略卡片/策略名.md
```

**用途**：可转SPEC的策略设计，给Hermes评审用的正式文档。
**模板**：`knowledge_base/98_模板/策略卡模板.md`

**必须包含**：
- 策略名称/编号
- 选股逻辑（条件、因子、权重）
- 买卖规则（入场、出场、仓位）
- 风险控制
- 回测结果摘要
- 状态标签（#待回测 / #已回测 / #可转SPEC / #已废弃）

### 2.3 回测记录（知识库 → 40_回测与实验）

```
knowledge_base/40_回测与实验/YYYY-MM-DD_策略名_回测结论.md
```

**用途**：记录每次回测的结论、参数实验、失败复盘。
**模板**：`knowledge_base/98_模板/回测记录模板.md`

**必须包含**：
- 回测日期、策略版本、commit
- 回测参数（时间范围、股票池、费率）
- 核心指标（年化收益、最大回撤、夏普比率、胜率、盈亏比）
- 与基准对比
- 结论（可实盘/需优化/已证伪）
- 状态标签（#待回测 / #已回测 / #已废弃）

### 2.4 回测报告（工程区 → trade_reports/）

```
trade_reports/
├── trade_report_YYYY-MM-DD.md     ← 详细回测报告
├── dashboard_YYYY-MM-DD.html      ← 可视化看板
└── deepseek_backtest_*.md         ← 大模型辅助回测报告
```

**用途**：机器可读的详细回测报告，含图表。
**规则**：回测脚本自动生成，一般不需手写。

### 2.5 回测成果总表（知识库 → 90_索引MOC）

```
knowledge_base/90_索引MOC/回测实验总表.md
```

**用途**：所有回测成果的索引清单。每次跑完回测后同步更新。
**记录内容**：日期、策略、参数、结论、报告链接。

### 2.6 策略卡片总表（知识库 → 90_索引MOC）

```
knowledge_base/90_索引MOC/策略卡片总表.md
```

**用途**：所有策略卡片的索引清单。每次新建/修改策略卡片后同步更新。

## 三、回测流程与产出物

### 3.1 独立脚本回测（新策略探索阶段）

```cmd
# 编写独立回测脚本
scripts/backtest_策略名.py

# 运行回测（用 nohup 后台跑）
nohup python scripts/backtest_策略名.py > trade_reports/output_YYYY-MM-DD.log 2>&1 &
disown
```

**产出物**：
- `trade_reports/trade_report_YYYY-MM-DD.md`
- `trade_reports/dashboard_YYYY-MM-DD.html`
- 如果用到大模型分析：`trade_reports/deepseek_backtest_*.md`

### 3.2 回测工厂回测（标准化组合回测）

```cmd
# 通过回测工厂配置跑批
python -m backtest.run_batch ...
```

**产出物**：
- `F:\backtest_workspace\策略名\` 目录下
- `F:\backtest_workspace\batch_summary\` 汇总

### 3.3 数据源

| 数据源 | 路径 | 覆盖范围 |
|--------|------|---------|
| astock（主数据源） | `E:/astock` | 买断1min 2009起+财务全量 |
| QMT自持 | `xtdata` API | 实时补充 |
| 慧策系统 | 辅助 | 备用 |
| 金策智算 DuckDB | 备用 | 已降级 |

### 3.4 工作日志

```
worklog/YYYY-MM-DD.md
```

每天的工作记录，非必须但推荐。

## 四、多Agent协作记录

当需要多个 Agent（CC / MIMO / RS / Hermes）协作时：

```
agent_hub/YYYY-MM-DD_主题名/
├── 00_brief.md           ← 任务说明
├── CC_review.md          ← CC 评审意见
├── Mimo_review.md        ← MIMO 评审意见
├── RS_review.md          ← RS 评审意见
└── Hermes_汇总.md        ← Hermes 终审汇总
```

**规则**：
- 每个主题独立目录
- 不要多人同时改同一个文件
- Hermes 终审后汇总到 `Hermes_汇总.md`

## 五、工程知识库

踩坑记录、QMT 操作、环境配置等，放：

## 六、标签体系

知识库文件使用以下标签：

| 标签 | 用途 |
|------|------|
| `#待整理` | 刚创建的草稿 |
| `#待验证` | 待验证的观点/发现 |
| `#已验证` | 已验证的结论 |
| `#已废弃` | 已过时或证伪 |
| `#可转SPEC` | 可转为策略设计规格 |
| `#待回测` | 等待回测验证 |
| `#已回测` | 已完成回测 |
| `#实盘观察` | 实盘中的观察记录 |

## 七、总结：一次完整策略研发的产出物清单

```
Step 1: 策略设计
  □ knowledge_base/30_策略卡片/策略名.md  ← 策略卡片
  □ specs/SPEC_策略名.md                   ← 策略规格（Hermes评审）
  □ Project_XX_策略名/项目总览与下一步.md   ← 项目总览

Step 2: 回测验证
  □ scripts/backtest_策略名.py              ← 回测脚本
  □ trade_reports/trade_report_*.md         ← 回测报告
  □ trade_reports/dashboard_*.html          ← 可视化看板
  □ knowledge_base/40_回测与实验/回测结论.md  ← 回测结论沉淀

Step 3: 索引更新
  □ knowledge_base/90_索引MOC/回测实验总表.md    ← 更新总表
  □ knowledge_base/90_索引MOC/策略卡片总表.md     ← 更新总表

Step 4: 代码落地
  □ adapters/qmt_wrapper.py  ← 策略逻辑
  □ config/global_config.yaml ← 配置
  □ scripts/qmt_test_*.py    ← 测试脚本

Step 5: 工程记录（可选）
  □ knowledge_base/60_工程知识库/  ← 踩坑记录
  □ worklog/YYYY-MM-DD.md         ← 工作日志
  □ agent_hub/YYYY-MM-DD_主题名/  ← 多Agent协作
```

## V1.0 — 流程重构：去 check_buy + 通达信公式 scan（2026-05-31）

### 核心理念修正
**之前错误的假设：** check_buy（买点1/买点2/双色带/MACD）是买入决策的核心。
**实际流程：** 通达信选股（7 条件）→ 池 → 8D 评分 → 买入。check_buy 是冗余门。

### 变更清单

#### 1. bypass check_buy
- `scripts/run_backtest.py:_patch_qmt_wrapper()` — 回测模式 check_buy 替换为始终返回 True
- 池中所有股票直通 8D 评分，不再被 check_buy 卡掉 ~77%

#### 2. 板块热度 mock
- `_compute_mock_sector_heat()` — 每 bar 基于池内近 5 日涨幅排序赋分
- top20% → 5-10 分 | middle40% → 1-4 分 | bottom40% → 0 分
- 注入 `_g_scorer.update_sector_bonus()`
- handlebar 内 `_run_sector_analysis` 返回空 dict，不覆盖已设热度（兼容性验证通过）

#### 3. scan_market 升级为通达信实战放宽版
- 删除原有轻量过滤（价格/ST/上市天数）
- 新增 `_tdx_formula_filter()` — 7 条件全量筛选：

| # | 条件 | 通达信原公式 | Python 等价 |
|---|------|------------|------------|
| 1 | 筹码密集 | COST(95)-COST(5) ≤ 25% | 60 日振幅 ≤ 25% |
| 2 | 突破密集顶 | C > REF(HHV(H,60),1) 三日内突破 | 同逻辑 |
| 3 | 蓄势 | 5 日 ≥ 2 天涨跌幅 < 3% | 同逻辑 |
| 4 | 多头排列 | MA5>10>20 + MA60 向上/走平 | 同逻辑 |
| 5 | MA5 角度 | ≥ 30° | 同逻辑 |
| 6 | 阳线 | C > O | 同逻辑 |
| 7 | 排除急拉坑 | 18 日坑幅 ≥ 16% + 3 日急拉 ≥ 13% | 同逻辑 |

#### 4. 代码清理
- `core/signal_main_rise.py`: 删除 check_buy / _filter_pit_breakout / 双色带 / N,N1,N2,N3,M,YANG_RATIO,CMF_PERIOD,CMF_THRESHOLD（-122 行）
- `adapters/qmt_wrapper.py`: 删除 `from signal_main_rise import check_buy, N, N1, N2, N3, M`
- `tests/test_signal.py`: 删除 TestCheckBuy
- qmt37_strategy/ 保留未删

  3. `_load_sector_heat()` / `update_sector_bonus()` 增加自动缩放 — 热度值 1-10 自动 ×10 映射到 10-100
  4. **三级兜底**：文件 → QMT API → `_calc_sector_heat_from_ohlcv()` 基于已加载K线近5日涨幅排序分段赋分（top20%→6-10, mid40%→1-5, bottom40%→0），零外部依赖，策略每次运行自动生效

## P0 H12 主升浪验证 — 暂停（2026-06-19）

### 背景

独立研究脚本（`research/main_uptrend/p0_h12_validation.py`），验证 H12 假设："主升浪开始后 60 日内涨 30%+ 概率显著高于随机"。绕开回测工厂 v0.2 MVP 直接读 huicexitong duckdb。

### 工程进展

| 工单 | 结果 |
|------|------|
| v1-v3 | 废弃，MIMO Edit-thinking 循环卡死 |
| v4 | 跑到 4000/7180 被关机中断（事后才发现是关机不是 stall） |
| v5 | 跑到 7/9 组被 90min timeout 掐 |
| v6 | **跑通 PASS**：CC 重写脚本拆 stage_a/b + parquet 缓存 + 预构建索引；stage_b 从 72min 降到 5.3min（14× 加速） |

### 报告"PASS"数据

默认组（rise=30%，dd=12%）：P=1.93%，W/L=9.43，候选 N=454,106。

## 卖出反查链路修复 — v1+v2 合入（2026-06-22）

### 现象

模拟账户 67014907 实测 2026-06-22 日盘，触发 600110.SH 底线清仓后：
```
[卖出委托] 600110.SH 400股 价格=16.84
[卖出清理] 600110.SH 订单号无效(0)，清理
```
8 个 bar 内挂 8 笔同价孤儿委托，没成交也没撤单。

### 根因（三环 bug）

1. `Trader._passorder` 是 QMT 异步接口，正常返回 `0` 或 `None`，**不是订单号**
2. `_check_and_execute_sell` 把 `0` 当 order_id 存入 `_g_pending_sells`（`is not None` 检查放过 0）
3. `_check_pending_sells` 用 `if not info.get('order_id'):` truthy 检查，把 `0` 误判成"无效"清理 → 策略以为没挂单，券商端还挂着，没人撤没人重挂

### 修复

`adapters/qmt_wrapper.py`：
- `Trader.sell`：passorder 调用后立刻反查 `get_trade_detail_data('order')`，返回真实 `m_nOrderID`，反查不到则返回 `None` 走"委托失败"路径
- `Trader._lookup_recent_order_id`（v2）：6 条 AND 过滤
  - code / volume / strategy_name in remark / status 排除 (54, 55, 57) 已撤已废
  - direction：`m_strOptName` 含 `卖`/`买`，字段空时**放宽不卡**（不一票否决）
  - t_before：`time.time()` epoch 转 HHMMSS，与 `m_strInsertTime[-6:]` 整数比较，-1 秒容差，解析失败时**放宽不卡**
- `_check_pending_sells`：`if not info.get('order_id'):` 改为 `if info.get('order_id') is None:`

### 流程教训

- **v1 静默简化**：v1 工单要求 6 条匹配条件，MIMO 实现 4 条（跳过 direction + t_before），但函数签名仍声明这两个参数，且自检表全 [x] 通过。CC 通过对照工单要求 grep 才发现。
- **v2 修复 + 流程升级**：
  - v2 把 direction/t_before 真正用起来
  - 自检表升级为"每条工单要求对应一条 grep"，杜绝"声明但不用 + 自检全勾通过"盲区
  - 增加 TASK-6 场景推演，要求 MIMO 在回执里写过滤逐条结果（不是只写 [x]）

## 二、已沉淀的关键教训

### 1. 初选 + 精筛不能机械写成同日 AND

来源：黄氏主升浪策略。

错误写法：

```text
combo_XG = box_breakout_XG AND double_zhongjun_XG
```

问题：箱体突破要求均线黏连、压缩末端；双中军要求多头排列、均线发散、动能确认。二者是先后阶段，不是同日并列过滤。

正确写法：

```text
box_window_hit = 最近 N 日出现 box_breakout_XG
combo_XG = box_window_hit AND 今日 double_zhongjun_XG
```

### 2. 形态看起来“最完整”不等于有正 alpha

来源：黄氏双中军 3 年回测。

双中军因覆盖趋势 + 动能 + 突破 + 大盘环境被评为 Top 1，但 2023-2026 中小盘回测显示信号胜率约 34%，完整工程持仓胜率约 30.7%，累计收益大幅为负。后续策略评审必须把“形态完整性”和“统计正期望”分开。

`stgentry.run_file()` 能跑生命周期和行情，但 `passorder` 不产生 `deal/order/account` callback，callback cache 为空。后续研究端回测继续走自建回测工厂；QMT GUI 回测只能作为人工辅助验证。

### 7. 卖出反查"查不到订单"≠委托失败，必须回查实际持仓

来源：2026-06-25 模拟盘 603618 杭电股份踩坑（commit 4140517 修复）。

C1 高位长上影减仓卖单 10:09 挂出（限价 56.28），QMT 回报 order stat=2（已报），但策略 `get_trade_detail_data('order')` 反查**查不到这条订单**，策略误判"委托失败、按失败处理"，`_g_my_codes`/卖出引擎没清。结果 14:30 该限价单实盘成交（56.37），账户归零，策略侧却残留 500 股，次日占第 3 名额卡住买入。

教训：卖出反查链路有三态——查到+成交、查到+未成交、**查不到**。前两态好处理，第三态最容易误判。`passorder` 返回 0/None（孤儿单）和"反查查不到订单"是两个不同 bug，[[sell-order-lookup-fix]] 记的 ab8f369 只修了前者，后者到 4140517 才补上。修法：反查失败时先 `_g_trader.get_position` 查实际 volume，归零=全成/减少=部成保留/未减=撤单重试，绝不直接误判失败撤掉活着的限价卖单。

### 8. 策略持仓记录必须和实盘账户强制同步，且不止买入窗口跑

来源：同 603618 案例。

策略 `_g_my_codes` 从文件读回，若上一交易日有卖出反查误判，残留的票会一直占名额。修法（commit 4140517）：新增 `_sync_holdings_from_account`，**开盘首帧 + 买入前双保险**用实际账户持仓强制同步 `_g_my_codes` 与卖出引擎 `current_shares`，已清的 pop+标 cleared；`可买数量` 用同步后实际持仓算，不靠会抖动的 `get_account_holdings`（同日卖出结算期 m_nVolume 会抖动，一会 0 一会 >0）。

教训：任何"策略记录 vs 实盘账户"的双份状态，都必须有强制对账机制，且对账时机要覆盖开盘首帧，不能只靠买入窗口那一次。
